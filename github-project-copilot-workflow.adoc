Single-page workflow for executing the active GitHub board with clear ownership, cadence, and quality gates.

== 1) Operating Objective

. Reduce production risk first.
. Keep work in small, mergeable increments.
. Preserve traceability from issue -> branch -> PR -> verification.
. Close issues only when acceptance criteria are validated.

== 2) Default Execution Model

Use this combined model unless a task requires a temporary override.

. Strategy: Two-Track Execution
.. Track A (Platform): #49, #55, #56, #58.
.. Track B (Product): #51, #50, #57, #52, #54, #53.
. Delivery style: Micro-PR Flow (target less than 200 LOC per PR when practical).
. Quality posture: Test-Guarded for #50, #51, #55, #56.

== 3) Collaboration Modes

Choose one mode per issue before implementation starts.

. Autonomous Executor (lowest interaction)
.. Human: sets scope and approves merge.
.. Assistant: implements and verifies end-to-end.
.. Use for low-ambiguity tasks.
. Manager + Delegate (light interaction)
.. Human: sets acceptance criteria and priorities.
.. Assistant: executes and reports midpoint/final status.
.. Use for medium-risk tasks.
. Pairing Loop (high interaction)
.. Human: makes tradeoff decisions in real time.
.. Assistant: iterates rapidly between decisions.
.. Use for architecture-sensitive tasks.
. Reviewer First
.. Human: writes code.
.. Assistant: performs bug-risk review and patch suggestions.
.. Use when branch ownership must stay human-led.
. Coach Mode
.. Human: implements directly.
.. Assistant: guides with step-by-step recommendations.
.. Use for learning-heavy tasks.

== 4) Mode Selection Rules

. Start with Autonomous Executor if scope is clear and under 2 hours.
. Switch to Pairing Loop when requirements are ambiguous or architecture is impacted.
. Switch to Reviewer First when a PR touches 3 or more subsystems.
. Use Coach Mode when learning speed is more valuable than delivery speed.

== 5) Daily Cadence

. Daily start (10 minutes)
.. Select one primary issue and one backup issue.
.. Confirm mode, acceptance criteria, and done conditions.
. Build block (60-120 minutes)
.. Implement one micro-PR maximum.
.. Run required checks before requesting review.
. Review block (20-30 minutes)
.. Review results and open risks.
.. Decide merge, rework, or split.
. Daily close (10 minutes)
.. Update issue with progress note and next action.

== 6) Weekly Cadence

. Monday planning
.. Commit to up to 3 in-progress issues total.
.. Prioritize by risk and dependency order.
. Midweek risk check
.. Resolve blocked items or de-scope quickly.
. Friday review
.. Verify done items include evidence and documentation updates.
.. Carry over only explicitly planned work.

== 7) Issue Lifecycle

. Ready
.. Acceptance criteria and scope are clear.
.. Dependencies are listed.
. In Progress
.. Branch created from issue number.
.. Collaboration mode selected.
. In Review
.. PR opened and linked to issue.
.. Verification evidence added.
. Done
.. PR merged.
.. Post-merge checks pass.
.. Issue comment summarizes outcome.

== 8) PR Checklist (Minimum)

. Scope
.. Only issue-scoped changes included.
. Verification
.. Lint passes.
.. Tests pass (or documented reason if not applicable).
.. Build passes for deployment target.
. Evidence
.. Include command output summary and any screenshots when UI changes.
. Traceability
.. PR body includes closes #issue-number.
. Risk
.. Include rollback notes for non-trivial changes.

== 9) Definition of Done by Issue Type

. Bug fix
.. Reproduction documented.
.. Fix verified.
.. Regression test added where practical.
. Architecture/flow
.. Decision recorded.
.. Conflicting paths removed or deprecated.
.. Docs updated.
. CI/testing
.. Pipeline green on PR.
.. New checks documented.
.. Failure modes explained.
. Refactor/tech debt
.. Behavior unchanged.
.. Complexity reduced and verified.
.. No new lint/type/build warnings in touched area.
. Docs
.. Commands and workflow are runnable as written.
.. Docs reflect current stack and deployment model.

== 10) Blocker Escalation Protocol

. Soft block (less than 30 minutes)
.. Assistant proposes 2 options with tradeoffs.
. Hard block (30-120 minutes)
.. Pause implementation.
.. Human decides option and updated scope.
. Critical block (more than 120 minutes or high-risk)
.. Split issue.
.. Land safe partial changes.
.. Open follow-up issue with explicit blocker notes.

== 11) Communication Contract

. One active issue at a time unless explicitly parallelized.
. Every update must include:
.. what changed
.. what was verified
.. what remains
. Human approval required before closing issue or merging high-risk PR.
. End each issue with a 3-line retrospective:
.. keep
.. change
.. stop
